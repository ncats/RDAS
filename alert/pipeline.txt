RDAS Alert Pipeline Workflow
============================

Entry Point
-----------

Main runner:

    alert/main.py

Primary class:

    AlertPipelineRunner

The runner owns the high-level order of the alert workflow. Individual task
classes extend PipelineBase, open the database connections they need, process
their work, and close their own connections.

The __main__ block in alert/main.py defines seven top-level steps with
run_step_with_timing(). At the time of writing, those step calls are commented
out and the script ends with pass. To run the full workflow from main.py,
uncomment the desired steps in order.


Core Runtime Pattern
--------------------

Each pipeline task follows this shape:

    1. __init__()
       Opens MySQL and/or Memgraph based on PipelineBase flags.

    2. find_new_data(gard_node)
       Used only by discovery tasks that search new data for a single GARD node.

    3. process_new_data()
       Used by staging, enrichment, graph, follow-up, person, and wrap-up tasks.

    4. close()
       Closes MySQL, clears Memgraph handle, and closes task logger handlers.

AlertPipelineRunner._run_pipeline_task() logs task start/end time and closes
the task if it still owns open database handles.


Important Flags
---------------

The alert pipeline is incremental. Most stages select only rows marked as new.

Common MySQL flags:

    is_new = 1
        Row belongs to the current alert run and should be processed.

    is_new = 0
        Row has already been consumed by the pipeline wrap-up step.

    alert_sent = 0
        New row has not yet been included in a user alert email.

    alert_sent = 1
        Row has already been included in alert email delivery.

Common incremental tables:

    clinical_trial
    clinical_trial_unique
    publication_article
    publication_omim
    person_of_all_sources
    organization_location


Top-Level Step Order
--------------------

Step 1. Find new clinical trial and publication updates

    Method:
        run_find_new_clinical_trial_and_publication_updates()

    Tasks:
        pipeline_1_gard.task_gard_1.GardNodeNamesTask
        pipeline_2_clinical_trial.task_clinical_trial_1.NewClinicalTrialDiscoveryTask
        pipeline_3_publication.task_publication_1.NewPublicationDiscoveryTask

    Flow:
        1. Load GARD nodes and their searchable names from Memgraph.
        2. For each GARD node, set the update date used for discovery.
        3. Search ClinicalTrials.gov for new matching trials.
        4. Search publication sources for new matching articles.
        5. Write newly found rows into update/staging tables.

    Notes:
        AlertPipelineRunner.TEST_LAST_UPDATE_DATE currently overrides the real
        GARD updated date. Set it to None for production behavior.


Step 2. Run MySQL database updates

    Method:
        run_mysql_database_updates()

    This method runs the clinical trial MySQL tasks first, then publication
    MySQL tasks.


Step 2A. Clinical trial MySQL staging and enrichment

    Method:
        run_clinical_trial_mysql_updates()

    Tasks, in order:
        1. NewClinicalTrialImportTask
           Copies newly discovered clinical trials into clinical_trial_unique.

        2. ClinicalTrialDrugInterventionMappingTask
           Extracts trial interventions, maps drugs to RxNorm, and stores
           intervention/drug mapping rows.

        3. ClinicalTrialPublicationMappingTask
           Extracts trial publication references and stores NCT/PMID mappings.

        4. ClinicalTrialPmidArticleImportTask
           Imports PubMed articles referenced by new clinical trials when those
           articles are not already present.

        5. NewClinicalTrialAnnotationTask
           Runs biomedical entity annotation over new clinical trials
           and stores UMLS annotation rows.


Step 2B. Publication MySQL discovery and enrichment

    Method:
        run_publication_mysql_updates()

    Tasks, in order:
        1. PublicationEpiNhsClassificationTask
           Classifies new articles as epidemiology and/or natural
           history studies.

        2. GardOmimPublicationMappingTask
           Builds GARD/OMIM publication mappings for new publication data.

        3. PublicationOminDataRetrievalTask
           Retrieves OMIM entry data needed for publication/OMIM processing.

        4. NewPublicationPubtatorRetrievalTask
           Retrieves raw PubTator data and parses it into structured rows.

        5. NewPublicationChemicalSubstanceTask
           Extracts chemical substance rows from new articles.

        6. PublicationFalsePositiveFilterTask
           Filters likely false-positive GARD/publication mappings.

        7. NewOmimPublicationArticleImportTask
           Retrieves/imports publication articles referenced from new OMIM data.


Step 3. Run Memgraph database updates

    Method:
        run_memgraph_database_updates()

    This method runs clinical trial graph tasks first, then publication graph
    tasks.


Step 3A. Clinical trial graph updates

    Method:
        run_clinical_trial_graph_updates()

    Tasks, in order:
        1. NewClinicalTrialGraphTask
           Creates ClinicalTrial nodes for new NCT IDs. Existing NCT nodes are
           not updated or duplicated.

        2. NewClinicalTrialGardRelationshipTask
           Creates ClinicalTrial-to-GARD mapping relationships.

        3. NewClinicalTrialConditionGraphTask
           Creates Condition nodes and links them to ClinicalTrial and GARD.

        4. NewClinicalTrialInterventionGraphTask
           Creates Intervention nodes and links them to ClinicalTrial.

        5. NewClinicalTrialDrugGraphTask
           Creates Drug nodes from RxNorm mappings and links them to
           Intervention nodes.

        6. NewClinicalTrialParticipantGraphTask
           Creates or reuses trial-specific Participant nodes.

        7. NewClinicalTrialPrimaryOutcomeGraphTask
           Creates or reuses PrimaryOutcome nodes for each trial outcome.

        8. NewClinicalTrialStudyDesignGraphTask
           Creates or reuses trial-specific StudyDesign nodes.

        9. NewClinicalTrialIndividualPatientDataGraphTask
           Creates or reuses trial-specific IndividualPatientData nodes.

        10. NewClinicalTrialAnnotationGraphTask
            Creates Annotation nodes from UMLS rows and links them to trials.


Step 3B. Publication graph updates

    Method:
        run_publication_graph_updates()

    Tasks, in order:
        1. NewPublicationArticleGraphTask
           Creates Article nodes from publication_article.

        2. NewPublicationArticleNodeAttrsUpdateTask
           Updates extra Article attributes such as full-text URLs, issue, and
           volume.

        3. NewPublicationEpidemiologyGraphTask
           Creates EpidemiologyAnnotation nodes and links them to Article.

        4. NewPublicationKeywordGraphTask
           Creates Keyword nodes and links them to Article.

        5. NewPublicationJournalGraphTask
           Creates Journal nodes and links Articles with published_in.

        6. NewPublicationMeshTermGraphTask
           Creates MeshTerm nodes and links them to Article.

        7. GardPublicationEpiNhsCountUpdateTask
           Increments GARD EPI/NHS article counters for new articles.

        8. NewPublicationGardArticleRelationshipTask
           Creates GARD-to-Article mentioned_in relationships.

        9. NewPublicationPubtatorGraphTask
           Creates PubtatorAnnotation nodes and Article annotation links.

        10. NewPublicationSubstanceGraphTask
            Creates Substance nodes and Article substance links.

        11. NewPublicationOmimRefGraphTask
            Creates OMIMRef nodes and Article OMIM reference links.


Step 4. Run pipeline follow-up updates

    Method:
        run_pipeline_followup_updates()

    Tasks, in order:
        1. GardRelationshipCountRefreshTask
           Refreshes relationship count properties on GARD nodes.

        2. ArticleGeneReviewFlagUpdateTask
           Downloads the GeneReviews title file and marks matching Article
           nodes with isGeneReview = true.

        3. OrganizationLocationRorLookupTask
           Finds Organization nodes without ROR data, calls the ROR API, and
           stages found/not-found results in organization_location with is_new=1.

        4. OrganizationLocationGraphSyncTask
           Applies new organization_location rows back into Memgraph by updating
           Organization nodes, creating Location nodes, and linking
           Organization-to-Location.


Step 5. Send alert emails

    Method:
        send_alert_emails()

    Task:
        alert_sender.AlertSender

    Flow:
        1. Read Firebase authenticated users and Firestore GARD subscriptions.
        2. For each subscribed GARD ID, count unsent new clinical trials and
           publications.
        3. Build an email payload for only the subscriptions with new data.
        4. Send an HTML alert email to the user.
        5. Send an operational summary email to configured admin recipients.

    Data sources:
        Firebase Auth
        Firestore subscriptions
        clinical_trial
        publication_article
        publication_gard_searchterm_pubmed_mapping

    Notes:
        alert_sender.py currently contains testing overrides for date range and
        recipient address. Review those before production use.


Step 6. Regroup people

    Method:
        run_regroup_the_person()

    Tasks, in order:
        1. NewPublicationPersonTask
           Extracts publication authors from publication_article and
           stages them in person_of_all_sources with is_new=1.

        2. NewClinicalTrialPersonTask
           Extracts responsible parties, central contacts, and officials from
           clinical_trial_unique and stages them in person_of_all_sources.

        3. NewGrantPersonTask
           Placeholder/empty grant person task.

        4. NewPersonGroupingTask
           Selects last names touched by new rows, fetches all people with
           those last names, runs PersonDisambiguator, and updates rdas_group_id.

        5. NewPersonAgentGraphTask
           Creates Agent nodes from grouped people and links them to Article,
           ClinicalTrial, Project, and Organization nodes.

        6. PersonPipelineWrapUpTask
           Resets person_of_all_sources.is_new to 0 after person graph updates
           are complete.


Step 7. Pipeline wrap-up

    Method:
        run_pipeline_wrapup()

    Tasks:
        1. ClinicalTrialPipelineWrapUpTask
           Resets clinical trial related is_new flags after the clinical trial
           pipeline has consumed the current-run rows.

        2. PublicationPipelineWrapUpTask
           Resets publication related is_new flags after publication graph and
           alert processing has consumed the current-run rows.


Recommended Full Run Order
--------------------------

Run the top-level methods in this order:

    1. run_find_new_clinical_trial_and_publication_updates()
    2. run_mysql_database_updates()
    3. run_memgraph_database_updates()
    4. run_pipeline_followup_updates()
    5. send_alert_emails()
    6. run_regroup_the_person()
    7. run_pipeline_wrapup()

This order preserves the expected dependency chain:

    discover new data
        -> stage/enrich MySQL rows
        -> update Memgraph nodes and relationships
        -> refresh derived graph metadata
        -> send alert emails
        -> extract/regroup people
        -> clear is_new flags


Operational Notes
-----------------

1. Each task has its own log file under logs/alert-<ClassName>.log.

2. The runner has its own log file:

       logs/alert-AlertPipelineRunner.log

3. Most graph tasks use MERGE to avoid duplicating existing nodes and
   relationships.

4. Wrap-up tasks should run only after downstream consumers have finished using
   is_new=1 rows.

5. If a task fails inside _run_pipeline_task(), the runner logs the error and
   re-raises it. The task is still closed in the finally block.

6. Discovery tasks use find_new_data(gard_node). Most other tasks use
   process_new_data().

7. PipelineBase.close() closes MySQL and logger handlers. The Memgraph handle
   is cleared by setting self.memgraph = None.


Directory Map
-------------

    alert/main.py
        High-level runner and step order.

    alert/pipelines/pipeline_base.py
        Base class for alert tasks.

    alert/pipelines/pipeline_1_gard/
        GARD node/name discovery.

    alert/pipelines/pipeline_2_clinical_trial/
        Clinical trial discovery, staging, graph updates, and wrap-up.

    alert/pipelines/pipeline_3_publication/
        Publication discovery, staging, graph updates, and wrap-up.

    alert/pipelines/pipeline_5_followup/
        Follow-up updates after graph data loads.

    alert/pipelines/pipeline_6_person/
        Person extraction, grouping, Agent graph sync, and person wrap-up.

    alert/firebase/
        Firebase Auth and Firestore subscription access.

    alert/emails/
        Email template rendering and SMTP delivery.

    alert/alert_sender.py
        Subscription matching and email orchestration.
